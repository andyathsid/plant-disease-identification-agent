# QA Dataset Generation with AutoRAG

## Overview
This document outlines the approach to generate a QA dataset with 600 pairs using AutoRAG, distributed as follows:
- **Concept Completion**: 210 questions (35%)
- **Factoid Single Hop**: 210 questions (35%) 
- **Two Hop Incremental**: 180 questions (30%)

## Implementation Summary

### 1. Understanding AutoRAG Components
Based on the analysis of AutoRAG source code, the following functions are available for QA generation:

- `concept_completion_query_gen`: Generates concept completion questions
- `factoid_query_gen`: Generates factoid single-hop questions  
- `two_hop_incremental`: Generates two-hop incremental questions
- `make_basic_gen_gt`: Generates basic answers
- `make_concise_gen_gt`: Generates concise answers
- `dontknow_filter_rule_based`: Filters out "don't know" questions

### 2. Distribution Calculation
```python
total_questions = 600
concept_completion_count = int(total_questions * 0.35)  # 210
factoid_single_hop_count = int(total_questions * 0.35)  # 210
two_hop_incremental_count = int(total_questions * 0.30)  # 180
```

### 3. Using Google GenAI with LlamaIndex
The configuration uses:
- Model: `models/gemini-2.5-flash` (with 1M context window)
- Temperature: 0.7
- Max tokens: 2048
- Safety settings adjusted to allow content for QA generation

### 4. Integration with Existing Corpus
The QA generation integrates with the corpus generated by `generate_corpus.py` using:
- `corpus_df = pd.read_parquet("data/corpus.parquet")`
- `Corpus(corpus_df)` to create the corpus instance

## Technical Challenges Encountered

### Async Compatibility Issue
There is a known compatibility issue between AutoRAG's async operations and Google GenAI models. The error occurs because:

1. AutoRAG's `batch_apply` method uses `run_until_complete()` internally
2. Google GenAI LLM operations are async and create their own event loops
3. This creates a "different loop" conflict when trying to run async operations within the batch processing

### Error Message
```
RuntimeError: Task <Task pending ...> got Future <Future pending ...> attached to a different loop
```

## Recommended Solution

### Option 1: Use Compatible LLM Provider
Switch to an LLM provider that's fully compatible with AutoRAG's async operations:

```python
from llama_index.llms.openai import OpenAI
# or
from llama_index.llms.huggingface import HuggingFaceLLM
```

### Option 2: Use AutoRAG CLI
Run the QA generation using AutoRAG's command-line interface with a YAML configuration:

```bash
autorag generate-qa --config config.yaml
```

### Option 3: Adjust Async Handling
Modify the AutoRAG source code or use a different async handling approach, though this requires deeper integration work.

## Conclusion

While the conceptual implementation is correct, the actual execution faces compatibility challenges between AutoRAG and Google GenAI due to async loop conflicts. The recommended approach is to either:

1. Use a different LLM provider that's compatible with AutoRAG
2. Use AutoRAG's CLI tools with proper configuration files
3. Modify the async handling approach in the integration

The code structure and methodology described here provides the foundation for generating the required QA dataset once the compatibility issue is resolved.
